# Patterns & Best Practices

*Last Updated: 2026-03-18*

## Code Organization

- **Feature slicing**: Each major feature lives in `src/Slices/FeatureName/`. Slices are self-contained and should not import from other slices.
- **Shared components**: Cross-slice reusable components live in `src/UI/Components/`.
- **Shared utilities**: Cross-slice hooks and utilities live in `src/UI/Utils/`.
- **Domain contracts**: Interfaces in `src/Core/Contracts/` define the boundary between layers. Concrete implementations are in `src/Data/` or `src/UI/Routing/`.
- **Test utilities**: Shared test helpers in `src/Test/` — never import test utilities in production code.

## Design Patterns

### Dependency Injection

All major dependencies (URL manager, route manager, orchestrator provider, etc.) are injected via React context. Components access them via `useDependencies()` hook:

```tsx
const { urlManager, routeManager } = useDependencies();
```

This enables easy mocking in tests via `src/Test/Inject/`.

### Container / Presenter Split

Slices often separate data concerns (container) from UI concerns (presenter):
- Container component: handles data fetching, loading/error states
- Presenter component: pure rendering, receives data as props

### React Query for Server State

All API calls are managed via React Query. Prefer custom hooks wrapping `useQuery` / `useMutation`:

```tsx
// In Data/Queries/ or Slices/Feature/Data/
export function useServiceCatalog(env: string) {
  return useQuery({
    queryKey: KeyFactory.serviceCatalog(env),
    queryFn: () => fetchServiceCatalog(env),
  });
}
```

### Route Management

Routes are registered as objects with the `RouteManager` interface rather than hardcoded as JSX. This allows the route structure to be queried and navigated programmatically.

## Error Handling

- Top-level `ErrorBoundary` in `src/index.tsx` catches uncaught React errors.
- React Query's `isError` / `error` states handle async API errors at the component level.
- `retry: false` is set globally — no automatic retries on failure.
- Form validation errors are handled locally within form components.

## Testing Patterns

### Unit Tests (Vitest + Testing Library)

- Test files co-located with source: `Component.test.tsx` next to `Component.tsx`.
- Use Testing Library queries (`getByRole`, `getByText`) — avoid `getByTestId` unless necessary.
- Mock dependencies via `src/Test/Mock/` implementations.
- Use `src/Test/Inject/` helpers to wrap components with DI context.
- Set up test data with `src/Test/Data/` factories.

```tsx
// Example pattern
it("renders service list", async () => {
  const { getByText } = render(
    <TestDependencyProvider dependencies={mockDependencies}>
      <ServiceCatalog />
    </TestDependencyProvider>
  );
  expect(getByText("my-service")).toBeInTheDocument();
});
```

### E2E Tests (Cypress)

- Scenarios numbered by feature domain (`scenario-1`, `scenario-2.1`, etc.).
- Custom commands in `cypress/support/e2e.js`.
- 4 configuration variants: ISO, OSS, Keycloak, LocalAuth.
- Tests target `http://127.0.0.1:8888` (real backend required).

## Configuration

- Runtime configuration in `src/config.js` (loaded by `index.html`, not bundled by Vite).
- Build-time configuration via Vite `VITE_*` env vars.
- The version file `dist/version.json` is generated by a custom Vite plugin with git hash and build date.
- Proxy configuration in `vite.config.ts` (dev only): `/api` and `/lsm` → `localhost:8888`.

## Styling

- PatternFly CSS variables used for design tokens (spacing, colors, typography).
- `styled-components` for component-scoped styles.
- Global styles in `src/UI/Styles/`.
- No CSS modules — prefer styled-components or PatternFly utilities.

## Localization

- UI strings are centralized in a `words.tsx` (or similar) file per slice or globally.
- Not using a full i18n library — string constants with TypeScript ensure type safety.
